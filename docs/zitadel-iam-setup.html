Zitadel IAM Setup — Self-Hosted Identity Provider

This document covers setting up Zitadel as an identity provider for clients who do not have their own SSO system (OneLogin, Okta, Azure AD). When a client already has enterprise SSO, use their provider directly — Zitadel is not needed.

1. Bridge Environment Variables

AUTH_MODE=oidc
OIDC_ISSUER=https://auth.prod.11mirror.com        # prod
# OIDC_ISSUER=https://auth.staging.11mirror.com   # staging
OIDC_CLIENT_ID=<oidc-app-client-id>
OIDC_AUDIENCE=<expected-aud-claim>

Set AUTH_MODE=builtin to disable external OIDC and use the bridge's built-in OAuth server (auto-approve flow, no external provider).

2. Installing Zitadel + Postgres

Infrastructure Path

Docker Compose file: infra/zitadel-auth/docker-compose.yml

The compose file runs two containers:

  • Postgres 16 — Zitadel's backing database
  • Zitadel v2.71.6 — the identity provider
Important: Use Zitadel v2.71.6, NOT latest (v4.x).
Zitadel v4.x ships login v2 as a separate Next.js application that is NOT bundled in the Docker image. Accessing /ui/v2/login/login returns {"code":5,"message":"Not Found"}. There is no way to make login v2 work without deploying the separate frontend container (which does not exist as a published image).

Zitadel v2.71.6 has login v1 built into the same binary — fully self-contained, no additional containers needed.

Docker Compose Configuration

Key environment variables in the compose file:

VariablePurpose
ZITADEL_MASTERKEYEncryption key (32 hex chars). Generate: openssl rand -hex 16
ZITADEL_EXTERNALDOMAINPublic domain (e.g., auth.staging.11mirror.com)
ZITADEL_EXTERNALPORT443 (Caddy handles TLS)
ZITADEL_EXTERNALSECUREtrue
ZITADEL_DB_PASSWORDPostgres password. Generate: openssl rand -hex 16
ZITADEL_ADMIN_PASSWORDInitial admin user password
DEPLOY_ENVstaging or production (determines data path)

Data path: /opt/deployments/${DEPLOY_ENV}/auth/pgdata

Port Mapping

ServiceContainer PortHost PortNotes
Zitadel808080908080 may be taken by other services (e.g., OpenShell)
Postgres54325433Avoids conflict with existing Postgres

Caddy Configuration

Add to the host's Caddyfile:

auth.staging.11mirror.com {
    reverse_proxy 127.0.0.1:8090
}

Caddy handles TLS via Let's Encrypt automatically. DNS A record must point to the EC2's IP.

Starting Zitadel

cd /opt/deployments/staging/auth
docker compose up -d

First boot runs start-from-init which creates the database schema and the initial admin user. Takes ~30-60 seconds.

Admin Login

After first boot, the admin username follows the pattern:

admin@zitadel.<external-domain>

For staging: admin@zitadel.auth.staging.11mirror.com

Zitadel Console URL: https://auth.staging.11mirror.com/ui/console/
Important: If you disable username/password login in the login policy (e.g., after adding Google), you will be locked out of the admin console. Always keep "Allow username/password" enabled as a fallback for the admin user.

3. Creating a Project and OIDC Application

3a. Create Project

  1. Log in to Zitadel Console
  2. Go to Projects → Create New Project
  3. Name it (e.g., "11mirror-agents")
  4. Note the Project ID — this becomes the OIDC_AUDIENCE value

3b. Create OIDC Application

  1. Inside the project → Applications → New
  2. Name it (e.g., "lolly-staging")
  3. Set Application Type: Web
  4. Set Authentication Method: PKCE
  5. Add Redirect URIs:
    • https://claude.ai/api/mcp/auth_callback (for CoWork)
    • https://lolly.staging.11mirror.com/auth-test (for our test page)
    • Any additional test URIs as needed
  6. Click Create
  7. Then it shows OIDC Configuration Under Grant Types, enable:
    • Authorization Code (default)
    • Refresh Token
  8. Save → note the Client ID — this becomes OIDC_CLIENT_ID

3c. Token Settings

  1. In the app settings, find Token Settings or Access Token Type
  2. Change from "Bearer (opaque)" to JWT
  3. Enable these checkboxes:
    • Add user roles to the access token — Yes
    • User roles inside ID Token — Yes
    • User Info inside ID Token — Yes
  4. Clock Skew — leave at 0 (default)
  5. Save

4. Configuring Google Login

4a. Google Cloud Console Setup

Create a project in Google Cloud Console (or use an existing one). The project has these key sections:

SectionPurpose
OverviewProject dashboard
BrandingAdd Authorized domains (e.g., 11mirror.com)
AudiencePublish the app — move from "Testing" to "In production" so external users can authenticate
ClientsCreate and manage OAuth 2.0 client credentials

Create OAuth Client

  1. Go to Clients → Create Client
  2. Application type: Web application
  3. Name it (e.g., "Zitadel Auth")
  4. Click into the created client and add Authorized redirect URI: 
    https://auth.staging.11mirror.com/ui/login/login/externalidp/callback
    (Replace staging with prod for production)
  5. Note the Client ID and Client Secret

Publish the App

  1. Go to Audience section
  2. Set User Type to External
  3. Change publishing status from "Testing" to "In production"
  4. This is required for external users (non-test accounts) to authenticate via Google

Authorized Domains

  1. Go to Branding section
  2. Add your domain (e.g., 11mirror.com) under Authorized domains

4b. Zitadel Console Configuration

  1. Go to Settings (gear icon) → Identity Providers → Google
  2. Paste the Google Client ID and Client Secret
  3. Click Optional, Configure auto-creation settings:
    SettingValueReason
    Automatic creationYesCreates Zitadel user on first Google login
    Automatic updateYesKeeps profile/email in sync with Google
    Account creation allowed (manually)NoHides the "Register" button
    Account linking allowed (manually)NoHides the "Link" button
    Determines whether an identity will be prompted to be linked to an existing accountDisabledDefault
  4. Click Create and then Save to activate the provider
  5. Now you will see this entry above the Add provider options. On mouse-over, it will show a tickicon. Click to Set as available

4c. Enable on Login Policy

  1. Go to Settings → Login Behavior and Security → Login Form
  2. Keep following as ticked/enabled. Rest should be disabled:
    • "Username and Password allowed" — admin fallback
    • External Login allowed
    • Domain Discovery allowed
  3. Save

6. Auth Test Page

A static HTML test page is deployed at:

https://lolly.staging.11mirror.com/auth-test

This page simulates the CoWork OAuth flow: PKCE authorization code → token exchange → POST /mcp with JWT. Use it to verify the full auth chain without needing a CoWork subscription.

7. Key Learnings

IssueResolution
Zitadel v4.x login v2 returns 404Use v2.71.6 — login v1 is built-in
ZITADEL_EXTERNALDOMAIN must match requestsInternal curl to 127.0.0.1 needs Host: header matching the external domain
sed -i breaks Docker bind mountsCreates new inode — must restart container after editing mounted files
Zitadel issues opaque tokens by defaultMust set app's access token type to JWT in token settings
Port 8080 conflictMap Zitadel to different host port (e.g., 8090)
Admin username formatadmin@zitadel.<external-domain>
Locked out after disabling password loginKeep "Allow username/password" enabled; if locked out, reset DB and reinit
Zitadel is event-sourcedDirect DB updates to projections are overwritten on restart — use the API or console
