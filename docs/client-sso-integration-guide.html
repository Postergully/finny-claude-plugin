Client SSO Integration Guide — OIDC Setup

What We Need From the Client

1. OIDC Application Configuration

Create an OIDC application named “Finny” in OneLogin. This is the same process followed for any SaaS application (e.g., NetSuite, Salesforce).

Item Description Example
OIDC Issuer URL OneLogin’s OIDC issuer endpoint https://acme.onelogin.com/oidc/2
Client ID The OIDC application’s client identifier a1b2c3d4-e5f6-...
Audience Expected aud claim in the JWT (typically same as Client ID, confirm if different) a1b2c3d4-e5f6-...

2. User Access Control

Assign users to the OIDC application in OneLogin:

  • Users who should access the agent → assigned to the app
  • Users who should NOT access the agent → not assigned (they will be denied at login)

This uses OneLogin’s existing role/group assignment workflow — no new process.

3. Token Configuration

Requirement Value
Access Token Type JWT (not opaque)
Signing Algorithm RS256
Required Claims sub, email, aud, iss, exp

4. OneLogin Application Settings

Requirement Details
Application Type Web
PKCE Support Must support S256 code challenge method
Discovery Metadata Must serve OIDC Discovery at .well-known/openid-configuration
Network Access Must be reachable from 160.79.104.0/21 (Anthropic egress range)
Token Endpoint Must accept Content-Type: application/x-www-form-urlencoded
Refresh Token Rotation Required for public client connections

What We Provide to the Client

1. Redirect URI

Add the following callback URL to the OIDC application’s allowed redirect URIs:

https://claude.ai/api/mcp/auth_callback

2. Required OIDC Scopes

The agent will request these scopes during authentication:

openid profile email

3. OAuth Discovery Endpoints

Our agent server exposes these discovery endpoints that MCP clients use to locate the OIDC provider:

GET <agent-server>/.well-known/oauth-authorization-server
GET <agent-server>/.well-known/oauth-protected-resource/mcp

MCP clients automatically follow this discovery chain — no manual configuration of auth endpoints needed by end users.

Integration Checklist

Security Notes

  • No client credentials (secrets) are stored on our side — PKCE replaces client secrets
  • Token validation uses OneLogin’s public JWKS keys (fetched and cached, refreshed on key rotation)
  • All communication over HTTPS
  • Tokens are validated for: signature, expiry, issuer, audience
  • User access revocation is immediate — removing a user from the app in OneLogin prevents new token issuance
