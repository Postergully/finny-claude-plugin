# Oracle redaction map

This file documents the redaction policy applied to `eval/oracle/*.json` so the
captured envelopes can be checked in to a public repo without leaking ShareChat
NetSuite data. The mapping is documented as **shapes**, never real values — the
runtime values that originally appeared in the envelopes are NOT recorded
anywhere in this repo.

## Applied substitutions

| Original token | Placeholder | Notes |
|---|---|---|
| `ShareChat` | `<company>` | Tenant name; appears in narrative envelopes |
| `MTPL` | `<entity-1>` | Subsidiary code |
| `finny-<uuid-v4>` | `finny-<session-N>` | Volatile session IDs; index `N` is per-file |
| `task_<id>_<seq>` | `task_<task-N>` | Volatile async task handles; index `N` is per-file |

## Placeholders the canonical queries already use

The capture-oracle helper sends inputs verbatim from `eval/canonical-queries.json`,
which already contains these stable placeholders. They appear in the captured
envelopes because the bridge echoes them back; they are NOT generated by this
redaction pass:

| Placeholder | Meaning |
|---|---|
| `<vendor-1>`, `<vendor-2>`, … | Vendor name |
| `<acct-N>` | NetSuite internal account ID |
| `<period-N>` | Period code (e.g. `Q1-FY26` shape kept where harmless) |
| `<entity-N>` | Subsidiary entity |
| `<amount-large>`, `<amount-small>` | Money amounts (none currently present) |

## How redaction is applied

```
node eval/redact-oracle.mjs eval/oracle
```

The script is idempotent (re-running on already-redacted files is a no-op) and
operates as raw-text substitution + per-file numbering of volatile IDs. It does
not parse envelope semantics, so it must be re-reviewed if the bridge starts
emitting a new PII-bearing field.

## Leak guard

The redaction PR run on CI must grep zero hits for:

```
grep -rEi 'sharechat|mtpl|kalicharan|11450275|finny-[0-9a-f]{8}-|task_[a-z0-9]+_[0-9]+' eval/oracle/
```

If this command produces any output, the redaction pass is incomplete.

## Known flaky queries

### q01-vendor-balance-discover

**Status**: drift-prone. Baseline accepts 19/20 pass; q01 is allowed to drift.

**Why**: The bridge does NOT short-circuit `phase: 'discover'` — it dispatches
the call to Finny's LLM via the gateway. Finny's discover-time output has three
legal shapes depending on upstream latency and her own response choice:

1. `status: 'partial'`, `data.shape: 'narrative'` — discovery narrative
   returned in time (this is the captured oracle)
2. `status: 'running'`, `data.value: task_<id>` — discovery exceeded the
   30s default `deadline_ms`, escalated to async
3. `status: 'error'`, `error.code: 'internal'` — gateway/transport
   transient failure

The captured oracle pins shape (1). Runs that produce (2) or (3) produce a
high-diff drift verdict, not a `pass`.

**Right fix (deferred)**: Implement a deterministic discover short-circuit in
`bridge/src/mcp/tools/query.ts`. When `phase === 'discover'` and the intent is
in the bless-list, synthesize a `needs_input` envelope from
`entry.required_scope` in-bridge — no LLM call. This matches the spec's stated
`expected_envelope_shape: 'needs_input'` for q01 and makes the eval a real
regression gate. Tracked as a follow-up to Phase 1 Task 1.3.

**Until then**: Phase 2 → Phase 3 gate accepts 19/20 pass as green, with q01
on a known-drift watchlist.
